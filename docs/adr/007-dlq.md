# ADR-007: Dead Letter Queue (DLQ) — Error Queue e Observabilidade de Falhas

| Campo | Valor |
|---|---|
| **Status** | Aceito (revisado Março 2026) |
| **Data** | Março 2026 |
| **Contexto** | Mensagens que excedem todas as tentativas de retry precisam ser preservadas de forma auditável e recuperáveis via processo controlado, sem perda de dados e sem reprocessamento duplicado. O plugin `rabbitmq_delayed_message_exchange` (anteriormente usado para `UseDelayedRedelivery`) foi descontinuado e arquivado em janeiro de 2026, com remoção planejada no RabbitMQ 4.3+. |
| **Decisão** | Utilizar a error queue nativa do MassTransit (`*_error`) como DLQ. Após esgotar os 5 retries imediatos (`UseMessageRetry`), o MassTransit move a mensagem para a error queue automaticamente e publica um evento `Fault<T>`. Um `TransactionFaultConsumer` consome esses eventos para registrar a métrica `cashflow.messaging.dlq_faults`, que dispara alertas operacionais no Azure Monitor. |

## Detalhes

### Topologia da DLQ no RabbitMQ

```
Mensagem falha (esgotou UseMessageRetry — 5 tentativas, 100ms-30s)
    ↓
MassTransit move para error queue: consolidation-transaction-created_error
    ↓
Publica evento Fault<ITransactionCreated> (automático)
    ↓
TransactionFaultConsumer consome Fault<T> → registra métrica cashflow.messaging.dlq_faults
    ↓
Azure Monitor alerta (Sev 2): "Dead Letter Queue Depth > 0"
```

### Observabilidade de Falhas

O `TransactionFaultConsumer` (em `Consolidation.API/Features/TransactionCreated/TransactionFaultConsumer.cs`) consome `Fault<ITransactionCreated>` e:

1. Registra log de erro com `MessageId`, tipo de exceção e mensagem.
2. Incrementa o counter `cashflow.messaging.dlq_faults` com dimensões `message_type` e `exception_type`.
3. A métrica é exportada via OpenTelemetry → Azure Monitor, onde o alerta KQL detecta novos faults em janelas de 1 minuto.

### Processo de Replay de Mensagens Mortas

1. **Identificação**: Alerta Azure Monitor detecta `cashflow.messaging.dlq_faults > 0`.
2. **Inspeção**: Acessar o RabbitMQ Management UI para inspecionar mensagens na fila `consolidation-transaction-created_error`. Headers `MT-Fault-*` contêm o stack trace e metadata.
3. **Correção**: Corrigir a causa raiz (bug no consumer, dado inválido no payload, schema incompatível).
4. **Replay controlado**: Reencaminhar mensagens da error queue para a exchange original via RabbitMQ Management API.

### Decisão de Remoção do `UseDelayedRedelivery`

O `UseDelayedRedelivery` do MassTransit dependia do plugin `rabbitmq_delayed_message_exchange`, que foi:
- **Arquivado** em janeiro de 2026 pelo time do RabbitMQ.
- Baseado em **Mnesia**, o data store original do RabbitMQ que será removido no RabbitMQ 4.3 ou 4.4.
- Uma **dependência operacional crítica** que falhava silenciosamente se o plugin não estivesse instalado.

A remoção simplifica a pipeline de resilience e elimina a dependência de um plugin descontinuado. A proteção contra falhas persistentes é mantida pelo `UseMessageRetry` (5 tentativas com backoff exponencial) seguido de encaminhamento automático para a error queue.

## Trade-offs

| Aspecto | Pipeline atual (sem delayed redelivery) | Pipeline anterior (com delayed redelivery) |
|---|---|---|
| Tempo até DLQ | ~30s (5 retries, 100ms-30s) | ~80min (5 retries + 3 redeliveries de 5/15/60min) |
| Dependência de plugin | Nenhuma | Plugin descontinuado `rabbitmq_delayed_message_exchange` |
| Operacional | Alerta imediato via `Fault<T>` consumer | Alerta após até 80min |
| Falhas intermitentes | Resolve em 5 tentativas | Resolve em 5 tentativas + 3 redeliveries |

## Consequências

- **Garantia**: Nenhuma mensagem é silenciosamente descartada. Toda falha persistente é preservada na error queue e gera alerta operacional via métrica `cashflow.messaging.dlq_faults`.
- **Detecção mais rápida**: Sem delayed redelivery, falhas persistentes são detectadas em ~30s em vez de ~80min. Isso permite correção mais ágil.
- **Sem dependência de plugin**: A pipeline de resilience depende apenas de funcionalidades nativas do MassTransit e RabbitMQ.
